V3 Restruct of Overall App

Context
- Primary Goal of DBPlanner: NoSQL-Designer should be able to store what we call a UniversalJSON json, let user make changes to it, and then export it to firebase,mongodb or cassandra as they see fit. 
- 3 Main Functions: Generate (or Upload a Schema json), Edit ( Schema further as User wish), export (it to one of the database formats).
- Our Goal here: Is to create the UniversalJSON to be used as storage for both Generate and Edit function. Export function will handle converting it specifically later. Do not confused this with the export function, export function is seperate, it wil take this schema and convert later. that is for later.

Goals for UniversalJSON.
- Be considerate for Export Function: Note the export function can only do so much, it should be able to convert the our UniversalJSON to mongodb/cassandra or firebase smoothly enough.
- Context: Heavy consideration must be taken here, to avoid big conflicts. UniversalJSON is purely for the website to handle, it just needs 2 key points
    1. Must be respectful to necessities asked by cassandra/firebase or mongodb as a nosql database.
    2. Must not conflict necessities (so export functino can work in peace).
    3. Does not have to implement all necessities if export function can do it on itself.
- Conclusion: Which requires the UniversalJSON to take consideration of all 3 database requirements at least subtly. 
- UniversalJSON considerations as of right now.
    - Adopt JSON Schema Standards: Align with JSON Schema conventions for consistency (e.g., required, $defs).
    - Add Key Definitions: Include a keys section for primary, partition, and clustering keys.
    - Clarify Nested Structures: Add a flag to indicate whether nested arrays/objects should be embedded (MongoDB/Firestore) or normalized (Cassandra).
    - Include Validation Rules: Define constraints like required, min, max, or pattern.
    - Support Database-Specific Extensions: Allow custom settings per database under an options key.
